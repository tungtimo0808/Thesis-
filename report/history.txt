================================================================================
1. GOAL (two task families on 924 panoramic dental X-rays)
================================================================================
A) Faster R-CNN object detection (built & trained, NOT the current focus):
     - FDI task     : detect + label each tooth by FDI number  (32 teeth).
     - Disease task : detect each tooth + label its condition   (12 classes).
B) Vision-Language report generation (CURRENT FOCUS):
     Fine-tune 5 open VLMs (LoRA) to read a panoramic crop and emit a STRUCTURED
     JSON dental report (per-tooth FDI + condition code + human name + a comment).
     All 5 models train on identical data and are compared head-to-head.


================================================================================
2. DATASET (PAN924) AND LABEL SPACE
================================================================================
Source annotations (in annotations/):
  dataset_final_v2.json        master dataset; input to the VLM builder.
  mouth_and_teeth_labels.json  raw COCO export, 20 categories (4 supercategories:
                               Mouth / Natural Teeth / Artificial Teeth / Mix Teeth).
  teeth_fdi_labels.json        FDI tooth-number labels.

RAW 20 -> CLEAN 12 disease classes (this trips people up; both the detector and
the VLM pipeline apply the SAME policy):
  (a) DROP 4 "Mouth" region labels (these are anatomy boxes, NOT per-tooth disease):
          Ed, De, Me, Mne
  (b) MERGE 4 raw labels into existing classes (config/disease_label_remap.json):
          Ri -> Te,  RiM -> Te,  TeM -> Te,  I -> M3i
  20 - 4 dropped - 4 merged = 12.

The 12 canonical disease/condition codes and their human names
(vlm_report_dataset/config/condition_map.json — authoritative "condition_name"):
  H    healthy                     R    restored
  C    caries                      Te   endodontic treatment
  Im   implant                     Rr   residual root
  M3i  impacted third molar        M3f  developing third molar
  CpuM prosthetic crown            Dc   crown destruction
  Di   incisal or occlusal wear    P    pontic
RARE classes (low support, the hard part of the task; oversampled later):
  Dc, Im, P, Rr, M3f   (train tooth-occurrences < ~160 each before rebalancing)

FDI detector uses 32 teeth -> num_classes 33 (+bg). Disease detector uses 12 ->
num_classes 13 (+bg). All COCO annotations are iscrowd=0 (each tooth is a single
object; the field only exists to satisfy the COCO schema).

4. FILE INVENTORY (what each script does + how to run it)
================================================================================
All scripts take --root (default ".") and are meant to be run from the project
dir. Paths below are relative to inredd-dataset-pan924-yolo-format-main/.

VLM pipeline (vlm_report_dataset/scripts/):
  build_5view_common_dataset.py  Build the model-agnostic "common" dataset from
        annotations/dataset_final_v2.json + images/. Splits by IMAGE 80/10/10
        (seed 924), crops each panoramic into 4 overlapping quadrants (overlap
        0.10) + keeps the full image = 5 views, writes common/images/<id>/<view>.jpg
        and common/{train,val,test}.jsonl + metadata/build_summary.json.
        Key args: --overlap-x/-y 0.10, --jpeg-quality 95, --train/val/test-ratio.
  generate_clinical_reports.py   Fill the free-text "comment"/"summary" fields
        (template-based) from common/raw/ into common/.
  rebalance_common.py            Oversample RARE conditions in TRAIN ONLY ->
        common_balanced/. Args: --in-dir common --out-dir common_balanced --cap 8
        --seed 924. Writes metadata/rebalance_summary.json.
  convert_dataset.py             Convert a common-format dir into the 5 per-model
        formats under converted/. Args: --input-dir (use common_balanced to get
        the oversampled train), --models qwen llava internvl phi paligemma.
        NOTE: this is why converted/qwen/train.jsonl has 5817 rows (from
        common_balanced) while common/train.jsonl has 3695.
  validate_common_dataset.py     Sanity-check the common dataset against
        config/condition_map.json. Writes metadata/validation_report.json.
  eval_report.py                 THE single metrics source (section 7).

VLM training tools (vlm_report_dataset/training/tools/):
  swift_pred_to_eval.py  Reshape `swift infer` output into {id, prediction} JSONL.
        Args: --val <gold jsonl> --swift-result <infer jsonl> --pred-field response --out.
  compare_models.py      Aggregate many metrics_*_test.json into a comparison table.
        Usage: python ... compare_models.py <glob of metrics_*_test.json>.
  validate_setup.py      Pre-flight environment check.

VLM training — RTX 3090 path (vlm_report_dataset/training/rtx3090/):
  shared_config.py       Single source of QLoRA hyper-params + helpers:
        train_model(model_name, output_dir, ...) builds & runs the `swift sft`
        command; evaluate_and_save(model_key, ...) scores every checkpoint on val
        by macro-F1, picks best, evaluates on test, writes results/<key>_results.txt.
        QLoRA 4-bit (bnb), MAX_PIXELS 802816, per_device 1 x accum 16, 2 epochs.
  train_qwen.py / train_internvl.py / train_llava.py / train_phi.py /
  train_paligemma.py     Thin per-model entry points calling shared_config.
  select_best_checkpoint.py, smoke_test.py  helpers.

VLM training — Colab A100 path (vlm_report_dataset/training/colab_a100/):
  generate_notebooks.py  *** SOURCE OF TRUTH. *** Generates the 5 notebooks from
        ONE template. Do NOT hand-edit the .ipynb — edit this file and re-run:
            <repo>\.venv\Scripts\python.exe generate_notebooks.py
        MODELS dict maps key -> (HF id, uses_MAX_PIXELS, gated, note).
  train_qwen.ipynb / train_internvl.ipynb / train_llava.ipynb / train_phi.ipynb /
  train_paligemma.ipynb  GENERATED (18 cells each). bf16 LoRA, MAX_PIXELS 1003520.
  README.md              How-to for the Colab path.

Faster R-CNN (faster_rcnn_fdi/scripts/ and faster_rcnn_disease/scripts/ — twins):

  frcnn_common.py, frcnn_dataset.py  shared model/dataset helpers.
  train_frcnn.py     Train from configs/train_config.yaml. Many --override flags

================================================================================

   ▼

   │  validate_common_dataset.py      (QA against condition_map)
   │
   ├─ rebalance_common.py  ──►  common_balanced/   (TRAIN oversampled; val/test same)
   │
   ▼  convert_dataset.py  (--input-dir common_balanced)

   │  swift sft  (ms-swift; uses converted/qwen for ALL models — see section 6)
   ▼
LoRA checkpoints  ─►  swift infer  ─►  swift_pred_to_eval.py  ─►  eval_report.py  ─►  compare_models.py

5 VIEWS per panoramic (924 images x 5 = 4620 samples = 4620 crop files):
  4 quadrant REGIONS (overlapping): image_upper_left, image_upper_right,
     image_lower_left, image_lower_right  -> task "regional_report" (regional prompt)
  1 "full" view (whole panoramic)         -> task "full_quadrant_report" (full prompt)

Split sizes:
  common (build_summary.json):   train 739 imgs/3695 samples · val 92/460 · test 93/465
  common_balanced (rebalance_summary.json): TRAIN 3695 -> 5817 rows (+2122 dup,
     +57.4%), val/test unchanged. Rare-class multipliers: Dc x7.89, Im x8.0, P x4.0,
     Rr x3.97, M3f x3.34 (regional-only oversampling, KG inverse-prevalence, cap 8, seed 924).
  converted/qwen (what training actually reads): train 5817 / val 460 / test 465.

IMAGE PATHS inside every JSONL are RELATIVE to the project dir, e.g.
  vlm_report_dataset/common/images/1119-F-59/image_upper_right.jpg
=> any runtime must chdir to the dir that contains vlm_report_dataset/ before training.

6. DATA SCHEMAS (concrete examples — what the bytes actually look like)
================================================================================
6.1 PROMPTS (vlm_report_dataset/config/prompt_templates/)
  regional_report_prompt.txt (one quadrant):
     "<image>\nThis is the {region} crop of a panoramic dental radiograph.\n
      Identify the visible annotated teeth using FDI notation and report each tooth
      condition.\nReturn only valid JSON with this schema:\n
      {\"region\":\"string\",\"teeth\":[{\"fdi\":\"string\",\"condition\":\"string\",
      \"condition_name\":\"string\"}],\"comment\":\"string\"}"
  full_report_prompt.txt (full image): asks for {"regions":{<4 regions each with
      teeth+comment>}, "summary":"string"}.

6.2 GOLD TARGET (assistant answer) — two schemas the eval understands:
  regional_report:
     {"region":"image_upper_right",
      "teeth":[{"fdi":"16","condition":"R","condition_name":"restored"}],
      "comment":"..."}
  full_quadrant_report:
     {"regions":{"image_upper_left":{"teeth":[...],"comment":"..."}, ...x4},
      "summary":"..."}

6.3 COMMON ROW (model-agnostic; one line of common/*.jsonl):
     {"id":"<imgid>_<view>_report","image":"vlm_report_dataset/common/images/<id>/<view>.jpg",
      "source_image":"<id>.jpg","view":"image_upper_right","task":"regional_report",
      "messages":[{"role":"user","content":"<image>\n..."},
                  {"role":"assistant","content":"{json target string}"}],
      "metadata":{"width":...,"height":...,"crop_boxes":{<region>:[x1,y1,x2,y2],...}}}

6.4 THE 5 CONVERTED FORMATS (convert_dataset.py builds these from a common row):
  qwen (ms-swift universal — USED FOR ALL 5 MODELS in the ms-swift path):
     {"id","messages":[{role:user,content:"<image>\n..."},{role:assistant,content:"{json}"}],
      "images":["...jpg"], "metadata":{...}}
  llava (HF multimodal chat-template):
     content is a typed list [{"type":"image","path":...},{"type":"text","text":...}]; no "<image>" token.
  internvl (ShareGPT):
     {"image","width","height","conversations":[{from:human,value:"<image>\n..."},{from:gpt,value:"{json}"}]}
  phi:
     {"image","prompt":"<|user|>\n<|image_1|>\n{prompt}<|end|>\n<|assistant|>\n","response":"{json}<|end|>"}
  paligemma:
     {"image","prefix":"{prompt no <image>}","suffix":"{json}"}

  *** ms-swift `swift sft --model <X> --dataset converted/qwen/train.jsonl` works for
      every model: ms-swift applies each model's own chat template internally based on
      --model. The other 4 formats are for training OUTSIDE ms-swift (HF/native scripts);
      the Colab notebooks do NOT use them. ***

6.5 COCO (detector annotations, faster_rcnn_*/data/annotations/*_coco.json):
  categories: disease -> 12 (+bg via num_classes 13); fdi -> 32 (+bg 33).
  every annotation iscrowd=0; one box per physical tooth (deduped by FDI).

7. EVALUATION (vlm_report_dataset/scripts/eval_report.py — THE metrics source)
================================================================================
Unit of evaluation = one tooth instance keyed by (sample_id, region, fdi); the
class is the condition code. Handles BOTH gold schemas (regional + full). Parses
predictions tolerantly (strips markdown fences / prose around JSON).

Run:
  # sanity (must be ~1.000):
  python vlm_report_dataset/scripts/eval_report.py --gold converted/qwen/val.jsonl --self-test
  # score predictions:
  python vlm_report_dataset/scripts/eval_report.py --gold converted/qwen/test.jsonl \
         --pred preds_test.jsonl --out-json metrics_qwen_test.json --tag qwen/test
  # preds line format: {"id":..., "prediction":"<json string or object>"}

Metrics produced (per_condition + aggregates):
  Detection (FDI): fdi_precision/recall/detection_f1, hallucination_rate (teeth
     invented), miss_rate (teeth missed).
  Classification (condition labels): per-condition P/R/F1+support, micro/MACRO/weighted.
     **macro_f1 is the HEADLINE metric** (robust to class imbalance; rare classes
     count equally) — checkpoint selection uses it.
  Joint correctness: condition_acc_on_detected, tooth_accuracy (detect+classify),
     exact_report_match_rate.
  Generation: json_valid_pct, text_rouge_l (self-contained LCS ROUGE-L on comments/
     summary), top_confusions (gold->pred on correctly-detected teeth).
  RARE set hard-coded = {Dc, Im, P, Rr, M3f}; printed/tiered separately.

Checkpoint selection (both training paths): score EVERY saved checkpoint on val
by macro_f1, keep the best, then evaluate it once on test. This generation step is
SLOW (can exceed training time) because it decodes the whole val/test set per
checkpoint — raising SAVE_STEPS reduces the number of checkpoints to score.
